## 1. 종합 판정 — 🔴 **RED (재작업 필요)**

**브랜드 일관성 점수: 5.5 / 10**

한 문장 요약: **"smalti"라는 이름 자체와 버벌·비주얼 산출물은 내부적으로 매우 정교하게 정합되어 있지만, 그 정합의 방향이 사용자가 명시적으로 요구한 해커 CLI 톤(jq/warp/zed/neovim)과 구조적으로 충돌한다.** 공예(craft)·비잔틴(Byzantine) 메타포는 그 자체로 아름답게 짜여 있으나 — 이것은 `etsy`·`substack`·`are.na` 계열 공방 브랜드의 결이지, 사용자가 요청한 터미널 네이티브·cryptic·hacky 톤이 **아니다**. 02번 문서에서 TOP 3(`nide`/`ravel`/`weavide`)가 사용자 톤 축에서 평가 27/30점으로 올라와 있었는데, 최종적으로 채택된 `smalti`는 **그 평가 매트릭스에 아예 등장하지 않은** 사후 선택이다. 채택 정당화가 어휘·서사 자산이 풍부하다는 이유에 치우쳐 있고, "해커 톤" 축에 대한 냉정한 점수 비교는 회피되어 있다.

판정 근거 3축:
1. **네이밍 의사결정의 점프컷**: 02 후보표의 TOP 5 중 어느 것도 smalti가 아니다. `rebrand-smalti.md` §8에서 "해커 톤 축 Partide 대비 우월"이라고 선언하나, `nide`(4자 cryptic, neovim 계보)와의 직접 비교는 누락.
2. **비주얼 팔레트 A(비잔틴 코발트·금·버건디)의 톤 이탈**: 해커 레퍼런스군(jq/warp/zed)은 **고채도 디지털** 또는 **무채색 + 단일 액센트** 라인이다. 비잔틴 금박·버건디는 **luxury craft** 레지스터 — Outlaw보다 Ruler/Creator 쪽 시각 코드에 가깝다. 04 §3.3의 자기 정당화가 논리적으로 성립하지만, **레퍼런스 대조**를 하면 납득되지 않는다.
3. **CLI 어휘 팩의 인지 부하**: `smalti chisel`, `smalti gild`, `smalti inlay`, `smalti refire` — 이 어휘들은 공예 레지스터 내에서는 우아하지만 **매일 타이핑하는 power user의 키보드 근육**에서는 번역 오버헤드를 만든다. `jq`/`rg`/`fd` 축의 미덕은 "3문자로 끝나는 동사" — smalti는 정반대로 간다.

## 3. 축별 평가

### A. 아키타입 정합성 — ⚠️ 부분 불일치

- **표면 정합**: 03 §4.1의 5축(공예/터미널 네이티브/cryptic/paradox/latent magic)은 Outlaw+Magician을 우아하게 구현한다.
- **실제 이탈**: 비잔틴 모자이크 메타포는 본질적으로 **Creator 아키타입** 영역이다. 장인·공방·배치·조립은 "새로운 것을 만드는" Creator의 언어. Outlaw는 "기성을 거부"이지 "새로 만들기"가 아니다. 지금 smalti는 Creator 60% + Outlaw 30% + Magician 10%에 가깝게 표현되고 있다.
- **팔레트 A의 문제**: 금박·버건디·코발트는 비잔틴 성당·황실 — **Ruler** 아키타입의 시각 코드다. 04 §3.3에서 "Outlaw의 깊이"로 변호하나, `warp`·`zed`의 실제 팔레트와 대조하면 Ruler 쪽으로 기운다.
- **결론**: 전략 문서의 아키타입 선언과 버벌·비주얼의 실제 구현 사이에 **정체성 드리프트(Outlaw→Creator/Ruler)** 가 있다. 이를 인정하고 방향 재설정하거나, Outlaw 톤으로 되돌려야 한다.

### B. 사용자 톤 정합성 — ❌ 구조적 불일치 (가장 심각)

사용자 톤 레퍼런스 대조표:

| 레퍼런스 | 핵심 질감 | smalti 현재 일치도 |
|----------|----------|:---:|
| `jq` | 2자, 설명 포기, `jq .` | ❌ (6자, 서사 과잉) |
| `oh-my-claudecode` | 자학·밈·prefix 패턴 | ⚠️ (`oh-my-smalti` 어색) |
| `warp` | 4자, 단음절, 속도감 | ❌ (6자, 속도감 없음) |
| `zed` | 3자, 대문자 없음, 단호 | ❌ (6자, 외래어 학습 필요) |
| `neovim` | `n-` 계보 선언, 재해석 | ❌ (공예 메타포로 카테고리 우회) |

**핵심 진단**: smalti의 결점은 "나쁜 이름"이 아니라 **"사용자가 요청한 결이 아닌 좋은 이름"** 이라는 것이다. 이것은 `substack`·`are.na`·`ghost.org` 같은 **크리에이터 툴** 브랜드 결에 가깝다. 02의 `nide`(4자, neovim 계보, ★★★★★ 해커톤) 또는 `ravel`(5자, 동사, ★★★★★ 해커톤)이 사용자 요청에 직접 부합한다.

### C. 어휘 팩 실용성 — ❌ 인지 부하 과다

| 일반어 | smalti | 인지 부하 | 실사용 예측 |
|--------|--------|:---:|----|
| `install` | `inlay` | 높음 | `npm install` 대비 번역 1단계 필요 |
| `uninstall` | `chisel out` | 매우 높음 | 2단어, 공예어, 즉각 이해 불가 |
| `publish` | `gild` | 높음 | "금박 입히기"를 매번 상기 |
| `major update` | `refire` | 매우 높음 | "가마 재점화" 메타포 과잉 |
| `minor update` | `polish` | 중간 | 일반 영단어라 수용 가능 |
| `registry` | `quarry` | 높음 | `npm registry`와 충돌 |
| `plugin` | `shard` | 중간 | 이미 DB sharding이 점유한 단어 |

**판정**: `jq`/`rg`/`fd` 생태계의 미덕은 "배우고 나면 0코스트" — 3자 명령은 한 번 익히면 끝이다. smalti의 공예 어휘는 **매 명령마다 은유 번역**을 강요한다. power user 대상 도구로는 치명적 단점.

**권고**: 이중 레일 **필수**. 공식 표층은 일반어(`install`/`uninstall`/`publish`), 공예어는 **별칭(alias)으로만** 제공. `smalti install` = `smalti inlay`, `smalti uninstall` = `smalti chisel`. 카피·스토리에서는 공예어가 빛나되, 실제 타이핑은 범용어로.

### D. 비주얼-버벌 엮임 — ⚠️ 부분 정합

- **정합 지점**: 컨셉 A(3×3 격자)는 `tessera` 어휘·모자이크 서사와 완전 일치. 컨셉 C(단일 타일)도 "smalti 1장 = 플러그인 1개" 정의와 정합.
- **불일치 지점**:
  - 팔레트 A의 화려함 vs 버벌의 "절제·소문자·마침표" 규칙 → 시각이 언어보다 한 톤 과함.
  - Tessellate-in 400ms 순차 모션은 **마케팅 랜딩** 톤이지 **터미널 네이티브** 톤 아님.
  - `i` 도트 이중색 처리는 작은 크기(favicon)에서 **소문자 `i` + 큰 사각 도트**가 `!` 또는 `¡`로 오독될 가능성.

### E. 리스크 재확인 — ❌ 미반영

- **Class 3 화장품 교차**: `rebrand-smalti.md` §6.3에서 경고했으나 03·04 어디에도 "출시 전 상표 clearance 필수" 명시 없음.
- **발음 분열(즈말티/스말티)**: 03 §4의 한국어 톤 가이드에 디스클레이머·공식 발음 가이드 페이지 링크 없음.
- **`smalti.com` 이탈리아 미술재료 상점 선점 가능성**: 실행 체크리스트에만 존재, 04·03에는 도메인 우선순위 반영 안 됨.

### F. 실행 가능성 — ⚠️

- **WCAG**: 자체 표 훌륭. `Cobalt Deep` 본문 금지 명시는 올바름. 단, `Gold Leaf`(#C9A24B)의 다크 배경 8.2:1은 맞지만 라이트 배경에선 **2.5:1 수준으로 탈락** — 04는 다크 기본이므로 큰 문제는 아니나 명시 필요.
- **폰트 라이선스**: 문제 없음.
- **8px 아이콘 식별성**: 컨셉 C의 "대각 이등분"은 이론적으로 식별 가능하나 **실측 렌더 증빙 0건**. Dock/taskbar에서 주변 네온 아이콘과 섞였을 때의 대비는 검증 필요.
- **`smalti` 워드마크의 `i` 도트 이중색**: 04 §10에서 본인도 실측을 요청. 작은 크기에서 **노이즈** 또는 **오탈자**처럼 보일 현실적 위험 존재.

## 4. 🔴 필수 수정

### R1. 네이밍 의사결정 재검토 [brand-strategist + naming-specialist]

- **현상**: 02의 평가 매트릭스 TOP 3(`nide` 27 / `ravel` 26 / `weavide` 25)가 해커 톤 축에서 우월함에도, 최종 채택 `smalti`는 이 매트릭스에 **등장하지 않음**.
- **지시**:
  1. `rebrand-smalti.md`의 평가 기준(어휘 자산·메타포 풍부함 등)을 02의 평가 매트릭스(독창성/발음/해커톤/네임스페이스/확장성/기억성)에 정렬하여 **같은 표 안에서 smalti vs nide vs ravel**을 1:1 비교하라.
  2. 특히 **해커톤 항목에서 smalti 점수를 냉정히 부여**할 것. 사용자가 명시한 레퍼런스(jq/warp/zed)와의 거리를 수치화.
  3. 비교 결과가 여전히 smalti가 승리한다면 그 근거를 명문화. 그렇지 않다면 smalti 채택 재고.

### R2. 사용자 톤 정합성 재선언 [brand-strategist]

- **현상**: 00_input.md의 "oh-my-claudecode나 jq 같은 hacky 톤" 요구가 smalti 선정 과정에서 **공예·서사 풍부함** 축으로 치환됨.
- **지시**: 01·02 사이에 누락된 "사용자 톤 최우선 원칙" 섹션을 복원하라. 모든 후보는 레퍼런스(jq/rg/warp/zed/nvim)와의 거리로 **먼저** 거르고, 그 다음에 서사·어휘 자산을 평가해야 한다. 현재 우선순위가 뒤집혀 있음.

### R3. CLI 어휘 이중 레일 의무화 [copywriter]

- **현상**: 03 §6의 서브커맨드 세트(`inlay`/`gild`/`chisel`/`refire`)가 단일 레일로 제시됨.
- **지시**: 모든 공예 동사에 일반어 **정식 별칭** 부여. 공식 문서·`--help` 출력은 일반어를 **표층**으로, 공예어를 **플레이버**로 이중 표기.
  ```
  smalti install <shard>  (alias: inlay)
  smalti remove <shard>   (alias: chisel)
  smalti publish <shard>  (alias: gild)
  smalti upgrade          (alias: refire)
  ```
  CLI 생태계 관습(`brew install`, `cargo install`, `npm install`)과의 키보드 근육 호환성 확보가 채택 조건.

### R4. 비주얼 팔레트 재검토 — A vs B 재평가 [visual-director]

- **현상**: 04 §3.3 권장 팔레트 A(비잔틴 코발트·금·버건디)는 어원 정합을 이유로 선택됨. 그러나 사용자 톤 레퍼런스(`warp`·`zed`·`jq` README) 팔레트와 **실제 거리가 멀다**.
- **지시**:
  1. 팔레트 A 정당화를 "어원" 한 축에서 평가했다면, "사용자 톤 레퍼런스"라는 **상위 축**으로 재평가할 것.
  2. 권장: **하이브리드 팔레트 C** 제안 — `ink.900/950` 중심의 warp/zed 다크 베이스 + 액센트 2개(`Gold Leaf` 1%, `Glass Cyan` 포커스)만 비잔틴에서 계승. Cobalt Deep과 Burgundy Tessera는 **surface·패널 레벨**로 강등(본문 등장 최소화).
  3. 만약 팔레트 A를 유지한다면, 근거가 "어원 > 레퍼런스"인 이유를 명문화.

### R5. 상표·도메인 clearance 선행 [identity-reviewer + 법무]

- **지시**: 03·04 문서 상단에 "⚠️ 본 아이덴티티는 Class 3/9/42 상표 실조회 통과 및 `smalti.dev`/`npm@smalti` 실선점 이전에는 외부 공개 금지" 블록 삽입. 런칭 게이트로 격상.

## 9. CLI 어휘 이중 레일 최종 권고 — **이중 레일 의무 채택**

```
표층 (일반 CLI 관습)            플레이버 별칭 (공예 레지스터)
─────────────────────────    ───────────────────────────
smalti init                  (동일)
smalti install <shard>       smalti inlay <shard>
smalti remove <shard>        smalti chisel <shard>
smalti list                  smalti mosaic
smalti search <query>        smalti quarry <query>
smalti publish <shard>       smalti gild <shard>
smalti upgrade               smalti refire
smalti create <desc>         smalti cut <desc>     ← cut만 표층 유지 가능 (coreutils 충돌 없음, 서브커맨드 네임스페이스)
```

- **--help 출력 원칙**: 표층 먼저, 우측에 `(aka: <별칭>)` 연한 회색으로 병기.
- **README 튜토리얼**: 표층 기준으로 작성. 공예어는 "있어도 좋은 플레이버"로.
- **공예어 생존 공간**: 릴리즈 노트 · 에러 메시지 · 커뮤니티 디스코드 · 로딩 스트링 — 이곳에서는 공예 레지스터가 밀도 있게 살아있어도 OK.

이중 레일 도입의 이점: 사용자 톤 요구(hacky, 관습 준수)와 브랜드 고유성(공예 서사) 양립. 한쪽을 포기하지 않아도 된다.
